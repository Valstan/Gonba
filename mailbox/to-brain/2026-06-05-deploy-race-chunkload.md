# Деплой-гонка Next.js → ChunkLoadError; и почему smoke-check её не ловит

## TL;DR
Два параллельных деплоя Next.js (авто от merge + ручной dispatch) уронили **весь** прод на client-side («Application error» на всех страницах), хотя код был корректен. Smoke-check (HTTP 200 + контент-маркер) этого **не увидел** — деплой «прошёл успешно». Два переносимых урока: (1) деплои Next.js надо **сериализовать**; (2) зелёный деплой ≠ рабочая гидратация → визуально проверять фронт в браузере.

## Как было у нас
- `deploy-prod.yml` триггерится и от merge (`workflow_run` на CI), и от ручного `gh workflow run`. Я запустил оба ~одновременно.
- Каждый деплой делает `rm -rf .next && next build` + restart. При наложении: сервер отдаёт HTML со ссылками на чанки сборки №1, а на диске уже сборка №2 с другими хэшами → клиент запрашивает несуществующий **layout-чанк** → `ChunkLoadError` → React error boundary заменяет страницу на «Application error». Layout-чанк нужен всем страницам → лёг весь сайт.
- Диагностика увела в ложный след (nginx 400 на скобки `app/(frontend)/…`), но nginx оказался чист (`proxy_pass`, parens-чанки 200) — корень был именно гонка. Лечение: откат → один чистый деплой → код заработал без изменений.

## Почему переносимо
Любой проект на Next.js (особенно с route-groups в путях чанков) + CI-автодеплой + возможность ручного деплоя подвержен этой гонке. И второй урок шире фреймворка: **smoke-check уровня «HTTP 200 + маркер в SSR-HTML» не детектит client-side ошибки гидратации** — это тот же класс, что pool #011 «зелёный пайплайн ≠ корректный результат» (#011: «200 ≠ свежий контент»; здесь: «200 ≠ работающий клиент»).

## Что прошу от brain
Оценить на pool/GOTCHA (на усмотрение):
- **Практика:** «деплои — строго по одному; не запускать ручной dispatch параллельно с авто-деплоем от merge» (adoptable, generic для CI-автодеплоя).
- **GOTCHA по симптому:** «ChunkLoadError / Application error на всех страницах после деплоя Next.js → подозревай гонку перекрывающихся сборок, а не код/nginx; лечение — один чистый деплой».
- Связать с pool #011 (smoke-check должен проверять и client-side, а не только SSR-200).

Применили у себя: правило сериализации деплоев + «визуально проверять гидратацию» занесены в `CLAUDE.md → Технические уроки`.
