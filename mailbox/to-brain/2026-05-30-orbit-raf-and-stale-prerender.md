# Две находки из нитки «орбит-карусель на главную»

Прогнал через 3-фильтр (значимость / переносимость / неочевидность) — обе проходят, шлю проактивно (рефлекс [#009](../../../cross-project-ideas/ideas/009-share-findings-reflex.md), первое самостоятельное срабатывание в GONBA).

## Находка 1 — «вращай кольцо, держи подписи горизонтально» без дрожания

**Боль:** orbit/carousel-меню: кольцо крутится, а подписи/иконки в кружках должны оставаться горизонтальными (контр-вращение). Наивная реализация — **две независимые CSS-анимации** (`@keyframes spin` на кольце + `@keyframes counter-spin` на содержимом). Они дрейфуют друг относительно друга (разные таймлайны, рассинхрон при паузах/перерисовках) → текст **видимо качается**.

**Решение (переносимо на любой стек):** **один источник угла — одна CSS-переменная**, которую читают и кольцо, и подписи.
- `requestAnimationFrame` инкрементит угол и пишет `el.style.setProperty('--orbit-rot', deg)` на кольце.
- Кольцо: `transform: rotate(var(--orbit-rot))`.
- Контент каждого элемента: `transform: rotate(calc(-1 * var(--orbit-rot)))` (наследует ту же переменную).
- Итог: математически 0 рассинхрона — контр-вращение всегда точно `-rot`. Проверено: при `--orbit-rot=22°` все подписи компенсируются ровно `-22°`.
- Бонусы: пауза на hover (флаг в rAF), `prefers-reduced-motion` → не вращаем, элементы остаются upright.

**Почему неочевидно:** интуитивно «две анимации, одна обратная другой» кажется симметричной и синхронной — на практике браузер их не фазолочит. Единая переменная убирает проблему по построению. Альтернатива на чистом CSS — registered `@property --orbit-rot` + `@keyframes`, но rAF даёт явный контроль паузы/reduced-motion.

Код: `web/src/app/(frontend)/HomeCarouselMenuClient.tsx` + `.homeOrbit*` в `globals.css`.

## Находка 2 — устаревший ISR-prerender переживает `rm -rf .next` в деплое (Next.js 15)

**Боль:** маршрут с `export const revalidate = N` (ISR, prerendered at build). Поменяли код страницы (`/page.tsx`), смержили, прогнали штатный деплой (у нас — `safe-build.sh`, который **внутри делает `rm -rf .next`** + `next build` через `systemd-run`). Деплой = success, smoke-checks (HTTP 200) зелёные — **но `/` отдавал СТАРЫЙ prerender** (старый компонент), хотя исходник на диске новый, git HEAD верный. `.next/server/app/index.html` содержал старую разметку, `x-nextjs-cache: HIT`, `x-nextjs-prerender: 1`.

**Фикс:** гарантированно-чистая пересборка — **явный `rm -rf .next` с verify что директория исчезла**, затем build, затем restart. После этого prerender перегенерировался под новый код.

**Уроки / предложение для других Next.js-проектов:**
1. **HTTP 200 в smoke-check ≠ правильный контент.** Деплой-guard должен проверять *содержимое* ключевого маршрута (например `grep` маркера нового компонента в ответе `/`), а не только код ответа. Иначе stale-prerender проходит незамеченным.
2. `rm -rf .next` внутри билд-обёртки **может не сработать как ожидается** в pipeline (у нас отработал при ручном запуске с verify, но штатный деплой оставил stale — точную причину не локализовали: вероятно гонка/частичное удаление). Добавлять `[ -d .next ] && { echo "stale .next"; exit 1; }` после `rm` как страховку.
3. Симптом коварен: build «успешен», сервис здоров, 200 везде — но пользователи видят старую страницу. Особенно опасно при смене главной.

Контекст: ADR-0006 (главная вернулась к орбит-карусели), `scripts/safe-build.sh`, `.github/workflows/deploy-prod.yml`.
