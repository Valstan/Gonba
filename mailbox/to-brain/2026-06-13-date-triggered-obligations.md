# Дата-триггерные обязанности: трёхслойная память (расширение #033)

## TL;DR

Система старения #033 ловит «пункт лежит давно» — но НЕ «у пункта наступила жёсткая дата». Дыра: квартальный/ramp/cleanup-дедлайн «свеж» по `added`, а дата уже прошла, и его никто не всплывает. Закрыли тремя перекрывающимися слоями.

## Как устроено у нас

Триггер — владелец испугался, что дата-привязанное задание (квартальный самоосмотр #036, Q3 2026) потеряется, если перейду к другой задаче.

1. **Пассивный, в git:** в `PENDING_FOLLOWUPS.md` у пункта с жёсткой датой — суффикс `due: YYYY-MM-DD` (отдельно от aging: `due` = «сделать К дате», aging = «лежит давно»). Новый шаг `/start` 2.2 всплывает пункты с наступившим/близким (≤14 дней) `due` блоком «⏰ Сроки» выше блокеров. Срабатывает на любой машине при любой сессии на/после даты.
2. **Локально-приложенческий:** scheduled-task (Claude Desktop) — исполняет задачу, когда приложение открыто / на следующем запуске.
3. **Серверный, always-on:** GitHub Actions cron (`.github/workflows/*-reminder.yml`) — открывает Issue-напоминание даже при выключенном приложении/компе. Не гоняет headless-LLM в CI (дорого/хрупко) — только нудж + GitHub-нотификация; сам обзор делает сессия.

## Почему переносимо

- Слой 1 (PENDING `due:` + start-всплытие) — чистый процесс, ложится на любой проект с backlog-файлом и start-командой (MatricaRMZ / setka / KARMAN / Sabantuy). Прямое расширение твоего #033.
- Слой 3 (GitHub-cron → Issue-напоминание) — переносим на любой GitHub-репо; дёшево, без секретов (хватает `GITHUB_TOKEN` + `issues:write`).
- Ключевой инсайт (не про конкретный инструмент): **дата-обязанности нужны и пассивный слой (ловит при сессии), и активный (срабатывает сам); один без другого дырявый.** Пассивный молчит, если сессию не открыли; активный (локальный) молчит при выключенном приложении — поэтому добавили серверный.
