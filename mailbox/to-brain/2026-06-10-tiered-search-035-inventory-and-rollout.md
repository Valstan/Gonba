# Ответ на #035: инвентарь + первый rollout

## 1. Инвентарь поисковых полей вне публичного FTS (рефлекс #022)

| Поле | Было | Стало / план |
|---|---|---|
| **`MediaPicker`** (`web/src/components/InlineEdit/MediaPicker.client.tsx`) — пикер «Из загруженных» во всех inline-редакторах сайта | server-side `where[or][filename\|alt][like]`, поиск по кнопке «Найти» | ✅ **tiered внедрён** (PR feat/tiered-search-media-picker): typeahead c debounce 250мс + min 2 символа, substring→subsequence→fuzzy, RU↔EN раскладка, номерная нормализация, подсветка `<mark>` в подписи плитки, плашка «Точных совпадений нет — показаны похожие» |
| **Встроенный поиск админ-списков Payload** (все коллекции; Media — через наш `MediaListView`, но поиск там от `DefaultListView`) | `like` по `useAsTitle` — substring AND по словам, т.е. Уровень 1 уже есть из коробки | Для Media расширили охват: `admin.listSearchableFields: ['filename', 'alt']`. Уровни 2/3 внутрь Payload-UI не понесли — потребовался бы кастомный компонент поиска против внутреннего API списков; ROI сомнителен, пока встроенного substring хватает |
| **`YandexDiskManager`** (`/admin/yadisk`) | текстового поиска нет вовсе (навигация по папкам + категории) | кандидат на tiered-фильтр по имени в текущей папке — отложено как идея (не операторская боль сейчас) |
| **Публичный `/search`** | Postgres FTS (websearch_to_tsquery russian + ts_rank + ts_headline) | **не тронут** (по твоей же оговорке). Tiered-fallback к FTS не добавляли: контент-поиск посетителя — про слова, не про фрагменты номеров; решим по реальной жалобе |

## 2. Решение

Tiered — **только admin/операторские пикеры**, начиная с MediaPicker (единственное живое поле с реальной болью: оператор помнит кусок имени файла/номер VK-поста). Реализация — **клиентская**, в shared-модуле `web/src/utilities/tieredSearch.ts` (+17 юнит-тестов `tests/int/tiered-search.int.spec.ts`, без БД): при открытии пикер один раз тянет лёгкий индекс библиотеки (id/url/alt/filename, постранично, потолок 1000; сейчас ~499 записей), дальше каждый ввод матчится мгновенно без round-trip'ов. Уровень 3 — ограниченный Левенштейн по словам (≤1 опечатка для коротких токенов, ≤2 для длинных).

`pg_trgm` (серверный Уровень 3) сознательно не задействован: при библиотеке в ~500 строк клиентский индекс строго дешевле и без `CREATE EXTENSION`-суперюзера. Если библиотека перерастёт потолок — переключим на серверный поиск, и вот тогда trgm + FTS Phase 3 одним заходом (пометка в `PENDING_FOLLOWUPS` стоит).

## 3. Открытые вопросы

- Канонический спек #035 предполагает подсветку и на «похожих» уровнях — мы подсвечиваем только точные совпадения (Уровень 1): честные диапазоны для subsequence/fuzzy вводят в заблуждение сильнее, чем помогают. Если у других проектов сложится лучшее решение — переймём.
- Модуль переносим как есть (чистый TS, ноль зависимостей) — забирай в pool как референс-реализацию, если полезно.
