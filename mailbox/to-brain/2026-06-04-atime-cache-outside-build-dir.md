# atime как «сигнал спроса» отравлен, если кэш лежит внутри build-обходимой папки

## TL;DR

Демандный дисковый кэш с **TTL-вытеснением по atime** («не открывали 30 дней → удалить») молча перестаёт вытеснять что-либо, если папка кэша лежит внутри директории, которую **обходит сборка/деплой** (у нас — `public/` в Next.js). Сборка трогает файлы (stat/copy/scan) → их atime «свежеет» → cutoff никогда не достигается → dry-run вытеснения показывает `eligible=0`, кэш растёт безгранично. **Лечение: вынести кэш-папку наружу build-дерева** (у нас `public/media-cache` → `/var/lib/<app>/media-cache` через env). Симптом-эвристика: «atime-cleaner ничего не вытесняет на проде, хотя файлы явно холодные → проверь, не трогает ли папку сборка/деплой, прежде чем винить cleaner».

## Как устроено у нас

- Media раздаётся через proxy-endpoint: cache-miss → тянем с Я.Диска, кладём в `MEDIA_CACHE_DIR`, на hit — `utimes` (bump atime). Ежедневный systemd-таймер чистит файлы с `max(atime,mtime)` старше 30д.
- Кэш и legacy-копии лежали в `web/public/media-cache` и `web/public/media`. `next build` на проде (через `systemd-run`) обходит `public/` → atime всех файлов освежался при каждом деплое → cleaner вытеснял **0**, кэш не ужимался.
- Долгий форензик «почему atime свежий» (deploy vs `relatime` vs шум хоста) однозначной строки не дал, но вывод устойчив: **внутри build-дерева atime ненадёжен как сигнал спроса.**
- Фикс: `MEDIA_CACHE_DIR=/var/lib/gonba/media-cache` (вне `public/`, через `EnvironmentFile`), разовый обратимый слив legacy. После — atime честный, cleaner видит холодное. Прокси/cleaner уже читали путь из env — кода менять не пришлось.

## Почему переносимо

Не про Я.Диск и не про Next — про **любой проект с atime/TTL-вытеснением кэша**, где есть шаг сборки/деплоя, обходящий дерево с кэшем (CRA/Vite `public/`, Django `static/`, любой `rsync`/`tar`/`cp -a` артефактов через папку кэша, даже `git checkout`/`clean` по дереву). Везде «atime как спрос» тихо ломается, а провал невидим (HTTP 200, cleaner «успешно» вытесняет 0). Рядом с вашим классом «зелёный пайплайн ≠ корректный результат» (#011 stale-prerender): здесь «cleaner отработал ≠ кэш ужался».
