# Build→CI gotcha: `ssh -f -N` форвард + немедленный pg_isready = гонка

Продолжение build→CI пайплайна, который ты принял 06-11 (G52 — lockfile). Нашёл вторую граблю того же шага «prerender читает прод-БД через SSH-туннель».

## Симптом

Деплой упал (1 из ~5 прогонов нового пайплайна) на шаге открытия туннеля:
```
ssh -f -N -o ExitOnForwardFailure=yes -L 15432:127.0.0.1:5432 GONBA
pg_isready -h 127.0.0.1 -p 15432   # → "127.0.0.1:15432 - no response"
```

## Почему это НЕ настоящий обрыв (важно для диагностики)

- SSH и Postgres были живы: on-failure-степ «дамп журналов прода» через **тот же SSH прошёл успешно** секундами позже; Postgres слушал `127.0.0.1:5432`; шаг 3× зелёный на 06-11.
- **Ре-ран деплоя сразу прошёл зелёным.**
- Корень: `ssh -f` форкается в фон сразу после установки форварда, а `pg_isready` стоит вплотную (без паузы) → иногда гонит готовность форварда. Класс «зелёный SSH ≠ готовый форвард» (родня pool #011 «зелёный пайплайн ≠ корректный результат»).

## Фикс (PR #139)

`pg_isready` в цикле до 10×2с вместо одиночной проверки. Дёшево (loop'ит только пока не готов), устойчиво к гонке форварда и медленному handshake. Проверено: следующий авто-деплой прошёл туннель чисто.

## Перенос

Build→CI с SSH-туннелем к живой БД — пока GONBA-специфика (Sabantuy прелендерит на пустой эфемерной БД, туннель не нужен). НО общий паттерн «ssh -f -N форвард в CI → проверять готовность с ретраями, а не одним немедленным probe» переносим на любой CI-туннель. Если build→CI с туннелем поедет на другие боксы кластера (KARMAN/Sabantuy при усложнении) — грабля приедет с ним. Предлагаю строкой в GOTCHAS рядом с G52 (симптом: «deploy падает на pg_isready/коннекте сразу после ssh -f -N, хотя SSH/БД живы»).
