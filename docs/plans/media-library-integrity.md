## Context

Владелец заметил: в Я.Облаке скопилась куча **дублей** картинок (каждая правка грузила файл заново). Запросы:
1. **Выбор из уже загруженных** вместо повторной загрузки (чтобы не плодить дубли). — ✅ частично сделано в этапе 1 (`MediaPicker`).
2. **Дедуп при добавлении** — не создавать дубль, если такой файл уже есть.
3. **Связи (usage) объекта** — знать, где картинка используется; при удалении предупреждать «используется, удалять нельзя», но разрешать принудительно; **лучше** — при удалении сразу предлагать выбрать замену, чтобы контент не осиротел.
4. **Инструмент слияния дублей** — если у картинки есть дубли и на них есть связи, оставить одну и перенаправить все связи на неё.

## Фундамент, который уже есть
- Media хранит `yandexSha256` (хэш содержимого) — готовый ключ для поиска дублей. Также `yandexResourceId`, `yandexPath`, размер.
- Раздача через `/api/media/file/[id]`; afterChange грузит на Я.Диск, afterDelete удаляет ресурс.
- `MediaPicker` (`web/src/components/InlineEdit/MediaPicker.client.tsx`) — выбор существующего файла (этап 1).

## Phase A — MediaPicker (✅ сделано, этап 1)
Кнопка «Из загруженных» в `InlineImage` и `EditProjectDialog`. Снижает появление новых дублей.

## Phase B — Дедуп при загрузке ✅ (2026-06-01)
- Реализовано на клиенте: `web/src/utilities/mediaUpload.ts` — `uploadOrReuseMedia(file, alt)` считает `sha256Hex(file)` (`crypto.subtle`), ищет `GET /api/media?where[yandexSha256][equals]=<hash>`; если дубль есть — переиспользует, иначе обычный `POST /api/media`. Я.Диск отдаёт sha256 содержимого, изображения не трансформируются (`imageSizes: []`) → хэш совпадает.
- Подключено в `InlineImage` и `EditProjectDialog` (upload-обработчики).
- **Ограничение:** дедуп только при загрузке через сайт-инструменты; старые дубли в Я.Облаке остаются — их разгребёт Phase D (слияние). Записи без `yandexSha256` (несинхронизированные) не матчатся — заливаются как обычно.

## Phase C — Связи (usage) + безопасное удаление

### C.1 — Usage-движок ✅ (2026-06-06)
- **Поиск использований** Media по id реализован: `web/src/server/media-usage/` (`sources.ts` — декларативная карта + `findMediaUsage.ts` — движок) + эндпоинт `GET /api/media/usage?id=<id>` (admin/editor/manager, read-only).
- **Ключевая находка** (интроспекция прод-производной БД 2026-06-06): media-ссылки в **rich-text** (Lexical upload-узлы `{type:'upload',relationTo:'media',value:<id>}`) Payload **НЕ** кладёт в `*_rels`-таблицы — они живут только в `jsonb`-колонке. Поэтому `payload.find` с `where` их не находит. Движок вместо этого гонит один параметризованный `UNION ALL` через `payload.db.pool` (как FTS Phase 1), покрывая: прямые FK-колонки, gallery/блок-массивы (1–2 hops по `_parent_id`), и рекурсивный JSONB-скан rich-text (`jsonb_path_exists(… '$.** ? (@.type=="upload" && @.value==$mid)')`). Карта из 27 источников по 9 коллекциям/глобалам — единственный источник правды, **новые media-поля добавлять туда** (иначе safe-delete осиротит контент).
- Только `$1` (media id, валидируется как int) динамичен → инъекций нет. Проверено на живых данных (media 305 = hero+meta+в тексте поста #26; 313 = галерея проекта #7; unused → 0). Юнит-тест `tests/int/media-usage.int.spec.ts` (9 кейсов, без БД).
- **Ограничение:** карта покрывает текущие (опубликованные/сохранённые) таблицы, не versioned `_*_v*` (draft-only ссылка не засчитается). Для safe-delete/merge на versioned-доках расширить карту `_v`-вариантами — отдельный шаг.

### C.2 — Безопасное удаление ✅ серверный хук (2026-06-06)
- `web/src/server/media-usage/preventDeleteHook.ts` — `beforeDelete` на Media зовёт C.1-движок; если media используется → `APIError(409, public)` с перечнем мест, удаление отменяется. **Защищает ВСЕ пути** (admin UI + REST), а не один виджет.
- **Escape hatch:** `context.forceDelete = true` обходит проверку (для серверных тулзов — Phase D слияние удаляет копию намеренно, перепривязав ссылки на канонную).
- Проверено end-to-end (delete используемого media → throw + файл на месте). Юнит-тест на `buildInUseMessage`.
- **Хвост (⏳ next, UI):** end-user force/replace из админки — кнопки «всё равно удалить»/«заменить на другую» (`MediaPicker` → перепривязать все ссылки на выбранную, потом удалить). Сейчас из admin UI used-media удалить нельзя (по дизайну, с понятной ошибкой) — что снимает «осиротевший контент».

## Phase D — Слияние дублей ✅ скрипт готов (2026-06-06), прод-прогон pending
- CLI `web/scripts/dedupe-media.ts`. Группирует Media по `yandexSha256`, канон = наименьший id (ранняя загрузка), перепривязывает все связи копий → канон (через usage-движок C.1), удаляет копии (+ Я.Диск-ресурс через afterDelete).
- **`--dry`** (по умолчанию, read-only) — план + сводка. Реальные числа: 118 групп, 212 копий (63 без ссылок, 149 со ссылками → 294 привязки).
- **`--apply`** — репойнт через **Local API** (`payload.update`/`updateGlobal`, версионно-корректно, `_status` сохраняется, синк поиска), карта `MEDIA_FIELDS`, рекурсивная замена Lexical upload-узлов. Инвариант: после репойнта `findMediaUsage(dup)===0` иначе abort; неизвестная коллекция → abort. Флаги `--keep-remote` (не трогать шаренный Я.Диск — обязателен на локали), `--group <sha>`, `--limit N`.
- **Провалидировано на локальном прод-дампе** (группа d48cbde3c592): FK (hero/meta) + rich-text (content) + vkq перепривязаны, копия удалена, search пересинкан, `_status` сохранён.
- **Прод-прогон (next, ДЕСТРУКТИВ):** после `pg_dump` + OK владельца, БЕЗ `--keep-remote`, по одному (`--group`/`--limit`). Исторические `_*_v` после удаления → FK `SET NULL` (мягкая деградация истории).

## Порядок и риски
- A (done) → B (дедуп, средний) → C (usage+safe-delete, крупный, ядро) → D (слияние, крупный, опирается на C).
- Главный риск — полнота карты «где используется media» (rich-text/блоки/массивы). Делать карту полей явной и тестировать на копии прод-БД, прежде чем что-то удалять/перепривязывать.
- Дедуп и слияние **не** запускать на проде без `--dry`-прогона и бэкапа (`pg_dump`).
