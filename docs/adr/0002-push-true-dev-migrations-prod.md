# 0002. Гибридная стратегия миграций: `push:true` на dev, миграции на проде

- **Status:** Accepted
- **Date:** 2026-04 (зафиксировано задним числом 2026-05-21)
- **Deciders:** Valstan

## Context

Payload CMS 3 поддерживает два режима эволюции БД через `@payloadcms/db-postgres` (Drizzle ORM):

1. **`push: true`** — на старте Payload сравнивает PostgreSQL-схему с TypeScript-коллекциями и применяет diff автоматически. Удобно для разработки (быстрый цикл).
2. **Файлы миграций** (`web/src/migrations/NNNN_*.ts`) — явные up/down скрипты, применяются через `pnpm payload migrate`. Контролируемо, версионируется в git.

На dev мы хотим **быстро менять коллекции** и сразу видеть результат — `push: true` идеально. На проде хотим **предсказуемость**: знать, что и когда применилось, иметь возможность откатить, не зависеть от стартового запроса.

Дополнительная проблема прода: `push: true` срабатывает на первом HTTP-запросе после старта. Если этот запрос прерывается прокси по timeout (нашему `gonba.service` бывало так после деплоя нового кода) — schema может остаться частично применённой.

## Decision

**Оставляем `push: true` в `payload.config.ts` для обеих сред** (упрощает локальную работу), **но на проде дополнительно применяем явные миграции** перед/после деплоя кода.

Concretely:
- На dev — `corepack pnpm dev` сам подгоняет схему (с интерактивным y/N от drizzle при существенных расхождениях; обходим через `yes y | ...` — см. memory `dev_schema_push_prompt`).
- На проде после `git pull` нового кода: `ssh GONBA "cd web && corepack pnpm payload migrate"` (или fallback `psql -f` для текстового SQL-зеркала миграции, если drizzle подвисает на y/N).
- Все миграции пишутся **идемпотентно** (`IF NOT EXISTS`, `DROP NOT NULL`, `pg_constraint` guard) — безопасно повторно прогонять.

## Alternatives considered

- **Только `push: true` везде** — простейший вариант, но опасно на проде: если первый запрос прерывается, схема в полусломанном состоянии. История изменений не сохраняется.
- **Только миграции, без push** — пришлось бы для каждого изменения коллекции на dev писать миграцию руками, что замедляет итерацию и провоцирует пропускать миграции для мелких правок.
- **Отдельный шаг pre-deploy hook в systemd** — добавит сложности оркестрации; миграция перед каждым стартом сервиса избыточна и замедляет рестарт.

## Consequences

### Положительные

- **Dev-loop остаётся быстрым** — изменил коллекцию, сохранил, перезагрузил dev-сервер, схема применилась.
- **Прод-схема предсказуема и версионируется** в `web/src/migrations/`.
- **Возможен откат** через `payload migrate:down`.
- **Идемпотентность** позволяет повторно прогонять миграции на любых средах.

### Отрицательные

- **Два источника правды** для схемы: то, что drizzle нагенерировал на dev через push, и то, что лежит в `migrations/`. Они могут разойтись (наш кейс 2026-05-20: 7 колонок добавили на проде ALTER TABLE вручную, а в `migrations/` файла не было — пришлось сегодня писать дозаказ). Лечится дисциплиной: после серьёзных правок коллекций — `pnpm payload migrate:create`.
- **drizzle y/N prompt** при подгонке схемы на dev — без TTY (например в headless CI) подвешивает процесс. Workaround: `yes y | corepack pnpm dev`.
- **На проде `payload migrate` тоже может попасть на drizzle prompt** (видели сегодня — `payload migrate` подвис на 40+ минут). Fallback: применять SQL-зеркало через `psql -f` и руками `INSERT INTO payload_migrations`.

### Нейтральные

- Новые миграции нужно явно регистрировать в `web/src/migrations/index.ts`.
- CI deploy (`.github/workflows/deploy-prod.yml`) **не запускает миграции сам** — есть safety net: если в коммите новые файлы `web/src/migrations/*.ts`, deploy фейлится ДО билда, ожидая что разработчик применил их вручную через `/sql`.
