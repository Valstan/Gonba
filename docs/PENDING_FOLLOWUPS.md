### Архитектура / Media

- **Довести ADR-0001 до конца: Media-коллекция → Яндекс.Диск как единственный источник правды, локальный сервер — только кэш.**

  Сейчас ADR-0001 говорит «Yandex.Disk — основное хранилище медиа», но на практике коллекция `Media` всё ещё использует стандартный Payload upload (локальный disk на VPS). Yandex.Disk используется параллельно — авто-галереи проектов + кастомный менеджер `/admin/yadisk`. Это дублирование и расхождение источников правды.

  **Scope** (масштаб):

  - Загрузка файла через админку Payload → загружается на Я.Диск + Payload хранит метаданные (URL, размер, тип, alt) + опционально кэширует thumbnail на VPS для быстрой выдачи.
  - Endpoint `/media/<id>` отдаёт файл: если в локальном кэше есть и свежий — отдаёт его (мгновенно), иначе — proxy с Я.Диска + сохраняет в кэш + отдаёт. Аналогично существующему `/yadisk-api/preview`, но шире.
  - Image-resize через `next/image` или sharp работает с локальным кэшем (Я.Диск даёт original).
  - Существующие записи Media — миграция: для каждой записи залить файл на Я.Диск (если ещё не там), сохранить URL, очистить локальный disk.
  - Удаление документа Media → удаление файла на Я.Диске (через afterDelete-hook).

  **Подходы** (выбрать):

  - **A. Payload Cloud Storage plugin + кастомный adapter для Я.Диска.** Идиоматично, но требует написать `clientUploads`/`generateURL`/`handleUpload`/`handleDelete`/`staticHandler` под Я.Диск API. ~3-5 дней работы. Самый чистый путь.
  - **B. Кастомный `beforeChange`/`afterDelete` хук в коллекции `Media` + хранение `cloudPath` в поле.** Быстрее (1-2 дня), но менее интегрировано с Payload upload UI. Превью в админке потребует ручной настройки.
  - **C. Полностью заменить коллекцию Media на собственный CRUD над Я.Диском** — без Payload upload вообще. Самое радикальное, ломает совместимость с relationship-полями в Posts/Projects/Pages (которые ссылаются на Media по ID).

  **Подводные камни:**

  - **`next/image` + удалённые URL** — нужен `next.config.js → images.remotePatterns` для `*.yandex.net` / `disk.yandex.ru`. Уже частично есть для текущих галерей — проверить.
  - **Кэш на VPS легко переполнить** (25 ГБ диска). Нужен LRU-evict или сроки хранения.
  - **Я.Диск rate limits** — для редко запрашиваемых файлов первая загрузка с Я.Диска медленная (см. ADR-0001 negative consequences). Кэш помогает, но требует прогрева.
  - **Все существующие записи Media** — миграция нетривиальная (десятки/сотни записей, файлы на VPS, рискованно делать вслепую).
  - **`beforeRead`-хуки в Posts/Projects ожидают URL** — нужно проверить как сейчас формируется `Media.url` и переписать на cloudUrl.

  **Что НЕ делать в этой сессии:** Это много работы и риск. Нужно отдельное планирование с пользователем, проба-концепта на одном test-документе, постепенный rollout. Записано как идея, чтобы не потерялось.
