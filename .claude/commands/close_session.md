# /close_session — финализация сессии GONBA

**Единственная команда закрытия сессии.** За один заход делает три вещи:

1. **Гарантирует, что вся работа на GitHub** — источник истины при работе на нескольких машинах (днём один комп, вечером другой). Коммитит и пушит все рабочие правки (код + доки) через PR-flow ([ADR-0002](../../../brain_matrica/adr/0002-pr-only-flow-no-direct-push.md)) и проверяет **жёстким sync-гейтом**, что дерево чистое и всё на `origin`, прежде чем считать сессию закрытой (pool #010).
2. **Сводит ветки в чистое состояние** — консервативный auto-merge готовых doc/process-PR (только с явного согласия — на GONBA merge в `main` = авто-деплой) + финальный `fetch --prune` / `pull main` / удаление смерженных локальных веток, чтобы следующая машина после `git pull` получила полный снимок и нить не рвалась ([pool #007](../../../brain_matrica/cross-project-ideas/ideas/007-close-session-auto-merge.md)).
3. **Фиксирует контекст** в `docs/SESSION_HANDOFF.md` (текущая нитка, следующий шаг, failed approaches), чтобы следующая сессия (на этом или другом компе после `git pull`) подхватила работу без пересказа.

> Запускается и естественными фразами: **«закрой сессию»**, **«заверши сессию»**, **«закрываемся»** — все они означают эту команду.

**Что НЕ делает:** не деплоит на прод, не build, не запуск тестов, не прод-операции. Для релиза/деплоя — [`/reliz`](reliz.md). Здесь — фиксация контекста + sync на GitHub.

## Шаг 1. Контекст

Собрать в одном параллельном блоке Bash:

- `git log --oneline -10` — что наделали
- `git log -1 --format='%H %s' -- docs/SESSION_HANDOFF.md` — когда последний раз обновляли handoff
- `git status --short` — что не закоммичено
- `git rev-parse --abbrev-ref HEAD` — текущая ветка
- `bash scripts/git_sync_check.sh --gate || true` — текущее состояние синхронизации с GitHub
- Прочитать текущий `docs/SESSION_HANDOFF.md`

Если `git_sync_check.sh --gate` уже даёт **exit 0** (дерево чистое, всё запушено), последний handoff-коммит — сегодня, и значимых изменений в сессии не было — скажи «всё уже на GitHub и handoff свежий, закрывать нечего» и завершись.

## Шаг 2. Уточнить у пользователя

Через `AskUserQuestion` — **один блок** с 1-4 вопросами максимум:

- **Какая текущая нитка?** Если из аргумента команды уже понятно — пропустить. Иначе — короткое описание (1-3 предложения).
- **Следующий шаг?** Конкретно: первое действие следующей сессии, file paths + команды.
- **Status:** `ACTIVE` (есть нитка) или `IDLE` (всё закрыто, ждём следующих задач)?
- **Failed approaches?** Только при `Status: ACTIVE` — были ли в этой сессии подходы, которые пробовали и отвергли (PoC не взлетел, обсудили и отбросили)? Если да — кратко перечислить с причиной отказа. Если нет — пропустить. (При `Status: IDLE` спрашивать не нужно — durable-уроки из предыдущих failed approaches graduate в `CLAUDE.md`/ADR при перезаписи handoff, эфемерные просто отпускаем — они остаются в `git log` + теле PR.)

Если пользователь ничего не передал и из истории сессии всё ясно — можно действовать без вопросов, **показав** Claude'у драфт и попросив подтверждения.

```markdown
**Branch:** <текущая ветка, обычно main>
**Last released version:** <если применимо, иначе ->

## Текущая нитка

<1-3 предложения о текущей теме>

## Контекст

- **План:** <путь к docs/plans/X.md если есть, иначе ->
- **Связанные коммиты сессии:** <hashes + одна строка о каждом>
- **Прод:** <статус>
- **Открытые вопросы для пользователя:** <список или нет>

## Failed approaches (этой нитки)

- **<подход X>** — попробовали в коммите [<hash>](...). Не сработало: <причина>. **Не повторять** без новой инфы.
- **<подход Y>** — отклонён на этапе обсуждения по причине <...>.

<если не было отвергнутых — оставить «_Не было — все попробованные подходы сработали или ещё в работе._»>

<eternal-уроки уровня проекта → CLAUDE.md, architectural alternatives → ADR в docs/adr/, post-mortem конкретной сессии → тело close-PR + git log (отдельного журнала нет, ADR-0007).>

## Не забыть (low-priority)

<точечно из PENDING_FOLLOWUPS или из памяти сессии>

---

```

**Важно про «Failed approaches»:**

- Секция **только для текущей нитки** (`Status: ACTIVE`). Если нитка закрывается (`Status: IDLE`) — секция убирается: durable-уроки graduate в `CLAUDE.md`/ADR, эфемерные отпускаем (остаются в `git log` + теле close-PR).
- **Grade'ы (где что фиксируем):**
  - `CLAUDE.md → Технические уроки` — **вечные уроки** уровня проекта (build только через safe-build.sh и т.п.)
  - `docs/adr/NNNN-*.md → Alternatives considered` — **архитектурные** отвергнутые альтернативы
  - `docs/SESSION_HANDOFF.md → Failed approaches` — **failed approaches в активной нитке** (живёт пока нитка идёт)
  - **тело close-PR + `git log`** — **session post-mortem** (когда нитка закрылась; отдельного журнала нет — [ADR-0007](../../docs/adr/0007-archive-development-log.md))
- При `/close_session ACTIVE` — спроси пользователя «были ли отвергнутые подходы за эту сессию?». При `Status: IDLE` — durable failed approaches из старого SESSION_HANDOFF подними в `CLAUDE.md`/ADR (если ещё не там), остальные отпусти.

## Шаг 4. Обновить PENDING_FOLLOWUPS.md (если нужно)

Если в сессии:
- Закрыли какие-то открытые задачи — отметить `✅ Сделано` (с PR/коммитом) или снять из списка (хронология — `git log` + тело PR).
- Появились новые техдолги / идеи — добавить с приоритетом 🔴⏳🟡🟢.
- Изменился приоритет существующих — обновить.

Не дублировать содержимое handoff'а — handoff ссылается, не повторяет.

## Шаг 4.5. Шеринг находки в мозг (условный)

Родилась ли в этой сессии значимая **переносимая** находка (новый скилл / фича / паттерн / решённая нетривиальная боль)? Прогони через 3-фильтр. Если выполнены **все три** — напиши письмо `mailbox/to-brain/YYYY-MM-DD-<slug>.md`. Если хоть одно «нет» — **ничего не пиши**. **Тишина = норма; по умолчанию молчим.**

**Фильтр (слать только если ВСЕ три выполнены):**

1. **Значимость** — новый скилл / фича / паттерн / решённая нетривиальная боль. **Не** рутинный фикс / бамп зависимостей / локальный рефактор / релиз.
2. **Переносимость** — применимо за пределами домена GONBA (другой проект — MatricaRMZ / setka / KARMAN — мог бы переиспользовать).
3. **Неочевидность** — без этой работы сам бы не знал; есть «урок», а не «сделал по инструкции».

Любое «нет» → молчим.

**Калибровка:**

- ✅ Делиться: удачный паттерн Payload-схемы, обход бага Next.js 15 (async `cookies()`/`headers()`), приём миграции Я.Диск/Media, CI/деплой-находка, обход бага инструмента, security-паттерн (как #008).
- ❌ Молчать: «поправил конкретную страницу гоньба.рф», «обновил Payload до X.Y», «отрефакторил наш VK-блок» — специфично для домена / рутинно.

**Формат письма** — `kind=idea`, frontmatter `from: GONBA` / `to: brain` / `date` / `topic` / `compliance` (`suggest`\|`recommend`) / `urgency`. Тело: **TL;DR** → **как устроено у нас** → **почему переносимо** → **что просишь от brain**. Образец — `mailbox/to-brain/2026-05-30-secrets-outside-repo-done.md` (секция adaptation notes). Писать **ТОЛЬКО** в свой репо (`mailbox/to-brain/`), коммитить через PR; brain прочитает на своём `/start` и оформит pool (асимметрия ADR-0001 v3). Это **исходящая идея** (не feedback на запрос) — поэтому `kind=idea`.

> Источник: директива brain `from-brain/2026-05-29-share-findings-reflex.md`, [pool #009](../../../brain_matrica/cross-project-ideas/ideas/009-share-findings-reflex.md). Цель — проекты сами делятся переносимыми фишками, а не только по явной просьбе.

## Шаг 5. Закоммитить и запушить ВСЁ (код + доки) через PR-flow

Цель — **ничего не оставить только на этой машине**. Прямой push в `main` запрещён (branch protection, [ADR-0002](../../../brain_matrica/adr/0002-pr-only-flow-no-direct-push.md)) — всё идёт через ветку + PR.

### 5a. Незакоммиченный рабочий код / доки (если `git status` непустой помимо handoff)

Если в сессии остались незакоммиченные правки кода или доков (не только handoff):

1. **Если на `main`** — создать feature-ветку:
   ```bash
   git checkout -b <type>/<slug>   # feat/ fix/ chore/ docs/ refactor/
   ```
2. Описательный commit (Conventional Commits, тело — что/почему/как применять на проде):
   ```bash
   git add -A
   git commit -m "$(cat <<'EOF'
   <type>(scope): <subject под 70 символов>

   <тело: что меняли, почему, как применять на проде — миграция? restart? ничего?>

   Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>
   EOF
   )"
   ```
3. `git push -u origin <branch>`.

Если кода значимо много / он сырой — спроси через `AskUserQuestion`, коммитить ли всё одним коммитом или разбить. По умолчанию цель — **запушить, не оставив рассинхрона** (push в feature-ветку снимает риск между машинами, даже если PR ещё не смержен; деплой — отдельно через `/reliz`).

### 5b. Handoff-коммит

Если ты на `main` и рабочих правок не было (только handoff) — создать `chore/handoff-YYYY-MM-DD`. Если уже на feature-ветке (5a или работали в ней) — добавить handoff-коммит **в ту же ветку**.

```bash
git add docs/SESSION_HANDOFF.md docs/PENDING_FOLLOWUPS.md
git commit -m "$(cat <<'EOF'
chore(session): handoff — <одна строка о нитке или «закрытие IDLE»>

<2-3 строки развёрнутого описания>

Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>
EOF
)"
git push
```

### 5c. PR

Если PR ещё нет — создать (`gh pr create`); если ветка с кодом уже имеет PR — handoff-коммит уже в нём. Merge — см. **Шаг 6** (консервативный auto-merge doc/process-PR с явного согласия) или вручную; **код-PR мержить только через `/reliz`** (контролируемый build+verify, т.к. merge = авто-деплой). **Не `gh pr merge --admin`**, не direct push в `main`.

## Шаг 6. Auto-merge готовых PR — Шаг A (pool #007, консервативно для GONBA)

> ⚠️ **GONBA-специфика: merge в `main` = авто-деплой на прод.** `deploy-prod.yml` триггерится на любой CI-зелёный push в `main` (`workflow_run` на CI, без path-фильтра). Поэтому auto-merge здесь **консервативный**: только doc/process-PR и только с явного согласия пользователя. Код-PR — всегда через [`/reliz`](reliz.md) (контролируемый build + verify).

**Что считается doc/process-PR** (безопасно auto-merge'ить — прод пересоберётся, но функционального изменения рантайма нет):
- изменения **только** в `docs/`, `mailbox/`, `.claude/`, корневых `*.md`.
- **НЕ** doc/process (→ только `/reliz`): любые правки в `web/`, `scripts/`, `deploy/`, `.github/workflows/`.

**Алгоритм:**

1. Собрать открытые PR и их состояние:
   ```bash
   gh pr list --state open --base main --json number,title,headRefName,isDraft,mergeable,reviewDecision
   gh pr checks <N>          # required-проверки (web-quality / CI) зелёные?
   gh pr diff <N> --name-only # какие пути трогает PR
   ```
2. PR — кандидат на auto-merge, только если выполнены **ВСЕ** условия:
   - base = `main`;
   - CI зелёный (все required-проверки прошли);
   - `mergeable = MERGEABLE` (нет конфликтов);
   - не draft;
   - содержимое **явно одобрено пользователем в этой сессии** (он сказал «merge» / «мержим»);
   - нет нерешённых review-комментариев;
   - **PR — doc/process**: `gh pr diff <N> --name-only` не содержит путей вне `docs/ | mailbox/ | .claude/ | *.md`.
3. Кандидатов собрать в **один** `AskUserQuestion` (не по каждому отдельно): список PR + явное предупреждение «merge запустит прод-пересбор (`deploy-prod.yml`)». Если пользователь уже явно сказал «мержи PR #N» в этой сессии — можно без повторного вопроса.
4. После подтверждения: `gh pr merge <N> --squash --delete-branch` (запомнить номера/ветки — пригодятся в Шаге 7).
5. **Код-PR** (трогают `web/` / `scripts/` / `deploy/` / `.github/`) — **не auto-merge'ить.** Перечислить в отчёте: «PR #N содержит код — merge через `/reliz`».
6. Если готовых doc/process-PR нет — шаг пропустить **молча**.

**Что НЕ делать на этом шаге:** не `gh pr merge --admin` (обход branch protection), не мержить без зелёного CI, не мержить код-PR в обход `/reliz`, не мержить draft / с конфликтами / с un-resolved review, не мержить PR, содержимое которого пользователь не одобрял в этой сессии.

## Шаг 7. Финальный sync локальных веток — Шаг B (pool #007)

Чтобы следующая машина после `git pull` получила полный снимок, а локальные ветки не копились.

```bash
git fetch --all --prune
# на main переходим только если рабочее дерево чистое (Шаг 5 уже всё закоммитил+запушил)
git checkout main && git pull --ff-only
git branch -vv     # увидеть локальные ветки и их upstream-статус
git status         # ожидаем clean
```

**Какие локальные ветки удалять (`git branch -D <branch>`):**

- **Ветки, которые сам auto-merge'нул в Шаге 6** — удалить (`--delete-branch` снёс только remote; локальная остаётся).
- **Ветки с пометкой `: gone]`** в `git branch -vv` (их upstream удалён при `--prune` ⇒ PR смержен и remote-ветка снесена) — подтвердить `gh pr view <branch> --json state -q .state` = `MERGED`, затем удалить.

> ⚠️ **Squash-merge gotcha:** GONBA мержит `--squash` ⇒ squash создаёт **новый** коммит на `main`, коммиты ветки **не становятся** его предками. Поэтому `git branch --merged origin/main` **НЕ** покажет squash-смерженные ветки, а `git branch -d` откажется их удалять. Признак смерженности при squash-flow — **не** `--merged`, а пара «номер из Шага 6» / «`: gone]` + PR `MERGED`». Удалять такие — `git branch -D` (force), это безопасно: работа уже в `origin/main`.

**Какие НЕ трогать:**

- Ветки с **un-pushed коммитами** или **без upstream** → оставить + перечислить в `SESSION_HANDOFF.md → Не забыть` (это WIP, возможно с другой машины).
- Ветки с **открытым (не смерженным) PR** → оставить (PR — их трекинг).
- Если после `git checkout main` дерево неожиданно грязное — **остановиться**, не форсить, разобраться (вернуться к Шагу 5).

## Шаг 8. Sync-гейт (жёсткий) — не закрывать пока ≠ 0

```bash
bash scripts/git_sync_check.sh --gate
```

Должно вернуть **exit 0** (дерево чистое + всё запушено на `origin`). Если **exit 1** — **НЕ объявлять сессию закрытой**: разобраться, что осталось незакоммиченным/незапушенным, дослать (вернуться к Шагу 5) и прогнать гейт повторно. Это защита против рассинхрона между машинами — пройти обязательно (pool #010).

> Гейт проверяет **текущую** ветку (после Шага 7 это обычно `main`): чистая + запушена + не ahead/behind. Это нормально:
> - Если doc/process-PR auto-merge'нут (Шаг 6) — `main` включает работу, гейт на `main` → `exit 0`.
> - Если осталась feature-ветка с **открытым** PR (код-PR → `/reliz`) — её работа уже **на GitHub** (запушена в Шаге 5), а гейт зелёный на `main`. Merge — отдельный шаг, не блокирует закрытие.
> - **Un-pushed WIP** на других ветках гейт на `main` не ловит — поэтому Шаг 7 их **оставляет и перечисляет в handoff** (`Не забыть`); запушенность всего рабочего обеспечивает Шаг 5.

## Шаг 9. Отчёт пользователю

Структура:

1. **Сделано в сессии:** 1-3 строки из git log.
2. **✅ Всё на GitHub:** ветка `<branch>` запушена, PR #N (смержен / ждёт ревью); `git_sync_check.sh --gate` → OK.
3. **Auto-merge / ветки** (Шаги 6-7): какие doc/process-PR auto-merge'нуты (или «кандидатов не было»); какие код-PR ждут `/reliz`; какие локальные ветки удалены (смерженные) и какие оставлены (WIP — продублированы в handoff).
4. **Handoff обновлён:** показать ключевые поля (Status, Текущая нитка, Следующий шаг).
5. **Что в PENDING_FOLLOWUPS изменилось** (если изменилось): закрытые / новые / переприоритизированные.
6. **Коммит / PR:** хеш + сообщение; «ждёт ручного merge: `gh pr merge <N> --squash --delete-branch`» или «PR содержит код — merge через `/reliz`».
7. **Готово к закрытию терминала / смене компа** — ничего не осталось только локально.

Финальная строка — «До следующей сессии. `/start` сам подсветит нитку (и предупредит, если что-то не на GitHub).»

## Когда НЕ использовать

- **Промежуточные паузы во время сессии** — handoff для перехода между сессиями, не для перерыва на обед.
- **Сессия без значимых изменений** — если ничего не меняли (только посмотрели логи), handoff обновлять не нужно. Команда мягко спросит и завершится без коммита.
- **Идёт активный релиз** (`/reliz`) — сначала доделай его, потом `/close_session`. Иначе handoff не отразит прод-состояние.

## Что НЕ делать

- **Не direct push в `main`** ([ADR-0002](../../../brain_matrica/adr/0002-pr-only-flow-no-direct-push.md)) — даже handoff идёт через ветку + PR.
- **Не деплоить на прод** (`ssh GONBA`, build, restart, миграции) — это `/reliz`.
- **Не auto-merge'ить код-PR** (трогают `web/` / `scripts/` / `deploy/` / `.github/`) в `/close_session` — merge = авто-деплой, идём через `/reliz`. Auto-merge (Шаг 6) — только doc/process-PR и только с явного OK пользователя.
- **Не удалять локальные ветки с un-pushed коммитами** (Шаг 7) — оставить + упомянуть в handoff. Удалять только полностью смерженные в `origin/main`.
- **Не объявлять сессию закрытой, пока `git_sync_check.sh --gate` не вернёт exit 0.**
- **Не дублировать `PENDING_FOLLOWUPS` в handoff** — handoff ссылается, не повторяет.
