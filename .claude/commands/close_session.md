# /close_session — финализация сессии GONBA

Задача: за один заход зафиксировать «что мы делали и что следующее» так, чтобы следующая сессия (или ты с другого компа) могла продолжить без пересказа.

**Что НЕ делает:** не релиз, не build, не прод-операции, не запуск тестов. Это инструмент **фиксации контекста**, не финализации релиза. Для релиза — `/reliz`.

---

## Шаг 1. Контекст

Собрать в одном параллельном блоке Bash:

- `git log --oneline -10` — что наделали
- `git log -1 --format='%H %s' -- docs/SESSION_HANDOFF.md` — когда последний раз обновляли handoff
- `git status --short` — что не закоммичено
- `git rev-parse --abbrev-ref HEAD` — текущая ветка
- Прочитать текущий `docs/SESSION_HANDOFF.md`

## Шаг 2. Уточнить у пользователя

Через `AskUserQuestion` — **один блок** с 1-4 вопросами максимум:

- **Какая текущая нитка?** Если из аргумента команды уже понятно — пропустить. Иначе — короткое описание (1-3 предложения).
- **Следующий шаг?** Конкретно: первое действие следующей сессии, file paths + команды.
- **Status:** `ACTIVE` (есть нитка) или `IDLE` (всё закрыто, ждём следующих задач)?
- **Failed approaches?** Только при `Status: ACTIVE` — были ли в этой сессии подходы, которые пробовали и отвергли (PoC не взлетел, обсудили и отбросили)? Если да — кратко перечислить с причиной отказа. Если нет — пропустить. (При `Status: IDLE` спрашивать не нужно — переноси предыдущие failed approaches в DEV_LOG, если были, при перезаписи handoff.)

Если пользователь ничего не передал и из истории сессии всё ясно — можно действовать без вопросов, **показав** Claude'у драфт и попросив подтверждения.

## Шаг 3. Обновить SESSION_HANDOFF.md

**Перезаписать целиком** по шаблону (история — через git, не аккумулируется в файле):

```markdown
# Session Handoff

**Status:** ACTIVE | IDLE
**Updated:** <сегодняшняя дата YYYY-MM-DD>
**Branch:** <текущая ветка, обычно main>
**Last released version:** <если применимо, иначе ->

---

## Текущая нитка

<1-3 предложения о текущей теме>

## Следующий шаг

<конкретно: первое действие следующей сессии>

## Контекст

- **План:** <путь к docs/plans/X.md если есть, иначе ->
- **Связанные коммиты сессии:** <hashes + одна строка о каждом>
- **Прод:** <статус>
- **Открытые вопросы для пользователя:** <список или нет>

## Failed approaches (этой нитки)

<если в активной нитке пробовали подходы и отвергли — фиксировать здесь, чтобы будущая сессия не повторила. Формат:>
- **<подход X>** — попробовали в коммите [<hash>](...). Не сработало: <причина>. **Не повторять** без новой инфы.
- **<подход Y>** — отклонён на этапе обсуждения по причине <...>.

<если не было отвергнутых — оставить «_Не было — все попробованные подходы сработали или ещё в работе._»>
<если Status: IDLE — всю секцию можно убрать (нет нитки → нет failed approaches).>
<eternal-уроки уровня проекта → CLAUDE.md, architectural alternatives → ADR в docs/adr/, post-mortem конкретной сессии → блок «Уроки» в DEVELOPMENT_LOG.md.>

## Не забыть (low-priority)

<точечно из PENDING_FOLLOWUPS или из памяти сессии>

---

> Sticky note — что было следующим шагом. Перезаписывается через `/close_session`.
```

**Важно про «Failed approaches»:**

- Секция **только для текущей нитки** (`Status: ACTIVE`). Если нитка закрывается (`Status: IDLE`) — секция убирается, а отвергнутые подходы переносятся в блок «Уроки» соответствующего блока `DEVELOPMENT_LOG.md`.
- **Grade'ы (где что фиксируем):**
  - `CLAUDE.md → Технические уроки` — **вечные уроки** уровня проекта (build только через safe-build.sh и т.п.)
  - `docs/adr/NNNN-*.md → Alternatives considered` — **архитектурные** отвергнутые альтернативы
  - `docs/SESSION_HANDOFF.md → Failed approaches` — **failed approaches в активной нитке** (живёт пока нитка идёт)
  - `DEVELOPMENT_LOG.md → блок сессии → Уроки` — **session post-mortem** (когда нитка закрылась)
- При `/close_session ACTIVE` — спроси пользователя «были ли отвергнутые подходы за эту сессию?». При `Status: IDLE` — перенеси failed approaches из старого SESSION_HANDOFF в DEV_LOG, если они там были.

## Шаг 4. Обновить PENDING_FOLLOWUPS.md (если нужно)

Если в сессии:
- Закрыли какие-то открытые задачи — снять их из списка (перенести в `DEVELOPMENT_LOG.md` если ещё не записано).
- Появились новые техдолги / идеи — добавить с приоритетом 🔴⏳🟡🟢.
- Изменился приоритет существующих — обновить.

Не дублировать содержимое handoff'а — handoff ссылается, не повторяет.

## Шаг 4.5. Шеринг находки в мозг (условный)

Родилась ли в этой сессии значимая **переносимая** находка (новый скилл / фича / паттерн / решённая нетривиальная боль)? Прогони через 3-фильтр. Если выполнены **все три** — напиши письмо `mailbox/to-brain/YYYY-MM-DD-<slug>.md`. Если хоть одно «нет» — **ничего не пиши**. **Тишина = норма; по умолчанию молчим.**

**Фильтр (слать только если ВСЕ три выполнены):**

1. **Значимость** — новый скилл / фича / паттерн / решённая нетривиальная боль. **Не** рутинный фикс / бамп зависимостей / локальный рефактор / релиз.
2. **Переносимость** — применимо за пределами домена GONBA (другой проект — MatricaRMZ / setka / KARMAN — мог бы переиспользовать).
3. **Неочевидность** — без этой работы сам бы не знал; есть «урок», а не «сделал по инструкции».

Любое «нет» → молчим.

**Калибровка:**

- ✅ Делиться: удачный паттерн Payload-схемы, обход бага Next.js 15 (async `cookies()`/`headers()`), приём миграции Я.Диск/Media, CI/деплой-находка, обход бага инструмента, security-паттерн (как #008).
- ❌ Молчать: «поправил конкретную страницу гоньба.рф», «обновил Payload до X.Y», «отрефакторил наш VK-блок» — специфично для домена / рутинно.

**Формат письма** — `kind=idea`, frontmatter `from: GONBA` / `to: brain` / `date` / `topic` / `compliance` (`suggest`\|`recommend`) / `urgency`. Тело: **TL;DR** → **как устроено у нас** → **почему переносимо** → **что просишь от brain**. Образец — `mailbox/to-brain/2026-05-30-secrets-outside-repo-done.md` (секция adaptation notes). Писать **ТОЛЬКО** в свой репо (`mailbox/to-brain/`), коммитить через PR; brain прочитает на своём `/start` и оформит pool (асимметрия ADR-0001 v3). Это **исходящая идея** (не feedback на запрос) — поэтому `kind=idea`.

> Источник: директива brain `from-brain/2026-05-29-share-findings-reflex.md`, [pool #009](../../../brain_matrica/cross-project-ideas/ideas/009-share-findings-reflex.md). Цель — проекты сами делятся переносимыми фишками, а не только по явной просьбе.

## Шаг 5. Commit + push

Один отдельный коммит **только** про handoff:

```bash
git add docs/SESSION_HANDOFF.md docs/PENDING_FOLLOWUPS.md
git commit -m "$(cat <<'EOF'
chore(session): handoff — <одна строка о нитке>

<2-3 строки развёрнутого описания>

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
EOF
)"
```

**Если ветка == main:** allow-rules в `~/.claude/settings.json` пропустят `git commit` и `git push origin main` (см. сессию 2026-05-22 про настройку классификатора). Просто `git push`.

**Если на feature-ветке:** обычный `git push -u origin <branch>`, потом пользователь сам мержит через `/reliz` или ручной PR.

## Шаг 6. Отчёт пользователю

Структура:

1. **Сделано в сессии:** 1-3 строки из git log.
2. **Handoff обновлён:** показать ключевые поля (Status, Текущая нитка, Следующий шаг).
3. **Что в PENDING_FOLLOWUPS изменилось** (если изменилось): закрытые / новые / переприоритизированные.
4. **Коммит:** хеш + сообщение.
5. **Готово к закрытию терминала.**

Финальная строка — «До следующей сессии. `/start` сам подсветит нитку.»

---

## Когда НЕ использовать

- **Промежуточные паузы во время сессии** — handoff для перехода между сессиями, не для перерыва на обед.
- **Сессия без значимых изменений** — если ничего не меняли (только посмотрели логи), handoff обновлять не нужно. Команда мягко спросит и завершится без коммита.
- **Идёт активный релиз** (`/reliz`) — сначала доделай его, потом `/close_session`. Иначе handoff не отразит прод-состояние.
